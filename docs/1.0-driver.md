# wp-init 1.0 Driver

This document is the working driver for moving `wp-init` from `0.10.x` to a stable `1.0`.

It captures the decisions made so far, the target architecture, the public API we want to stabilize, and the implementation sequence required to get there.

## Goal

Deliver a bootstrap library for modern WordPress plugin development that is:

- easy to start with
- explicit in behavior
- safe to extend
- compatible with PHP `7.4`
- suitable for broad use in WP Desk plugins

The library should stay small and fluid. It should help organize plugin bootstrapping without becoming a framework that forces a stiff application model.

## Design Principles

- Keep the public entrypoint tiny.
- Keep WordPress-specific behavior at the edge.
- Prefer explicit modules over magic detection.
- Fail fast on invalid user configuration.
- Recover only from operational failures where recovery is clearly safe.
- Preserve convenience for common cases without making the system fuzzy.
- Match WordPress reality instead of forcing generic abstractions where they do not fit.

## Decisions

### PHP Compatibility

`wp-init` must support PHP `7.4`.

Implications:

- runtime code must not use PHP 8-only syntax
- `#[...]` attributes must be removed from runtime code
- `mixed` type hints must be removed from runtime code
- any other PHP 8-only syntax in `src/` must be removed

Tests may use newer PHP when needed, but the distributed runtime library must remain parseable and executable on PHP `7.4`.

### Public API

The intended public API for `1.0` is:

- `WPDesk\Init\Init`
- configuration format
- service definition files
- hook definition files
- module registration and module configuration
- explicit boot gate abstraction

Everything else should be treated as internal unless intentionally promoted later.

In particular, current implementation types such as kernel internals, binders, loaders, and legacy adapters should not remain de facto public API.

### Bootstrap Style

The main plugin file should stay minimal:

```php
<?php
/**
 * Plugin Name: Example Plugin
 */

use WPDesk\Init\Init;

require __DIR__ . '/vendor/autoload.php';

Init::setup( __DIR__ . '/config.php' )->boot();
```

No additional bootstrap ceremony should be required for common use.

### Modules

The current `Extension` concept should be replaced with explicit `Module`s.

Modules are the unit for opt-in bootstrap features. A module may:

- register service definitions
- provide built-in hook bindings
- expose module-specific configuration
- contribute boot gates
- adapt legacy behavior

Modules should be explicit. They should not activate just because a package exists in the dependency tree.

### Module Configuration

Modules and module-specific config should live under one config key:

```php
'modules' => [
    \Vendor\Plugin\Module\RequirementsModule::class => [
        'requirements' => [],
    ],
    \Vendor\Plugin\Module\TrackerModule::class => [],
]
```

Rules:

- key is a module class string
- value is `array` or `null`
- `null` is normalized internally to `[]`
- module order is preserved by array order

This replaces a separate `module_config` bag.

### Protected Defaults

`wp-init` may keep a very small protected default module set, but it must stay minimal and generic.

WP Desk-specific behavior should not be silently enabled by default in the core package.

### Hooks vs Callables

The main integration unit remains a `Hookable` class.

Use `Hookable` when:

- something attaches one or more WordPress hooks
- behavior is conditional
- hooks are attached later
- hooks may be detached
- the bootstrap logic has state

Callable bindings remain supported, but only as a narrow convenience feature.

Use callable bindings for:

- one-shot boot bridges
- small migration calls
- simple integration shims

If a callable needs conditions, multiple steps, or lifecycle behavior, it should become a `Hookable`.

### Boot Gates

The current stoppable binder behavior should be replaced with an explicit boot gate abstraction.

Boot gates exist to decide whether plugin boot may continue.

A boot gate is not just another hook binding. Its purpose is different:

- check environment or dependency requirements
- stop further plugin bootstrap when required
- optionally render notices or perform failure handling

This is the official mechanism for:

- PHP requirements checks
- dependent plugin presence/version checks
- userland hard-stop conditions

### Activation and Deactivation

Activation and deactivation are special WordPress lifecycle cases and should be treated explicitly.

They should not be forced into the normal hook binding model.

`1.0` should provide a simple explicit way to support:

- activation
- deactivation

without inventing a generic lifecycle framework.

### Legacy Support

Legacy support should become an explicit module, not a root boolean mode.

Target direction:

```php
'modules' => [
    \WPDesk\Init\Module\LegacyBuilderModule::class => [],
]
```

Legacy behavior should be isolated, testable, and clearly documented as a migration bridge.

### Environment and Debug

Environment inference is useful, but its precedence must be explicit.

Target resolution order:

1. explicit config `environment`
2. `wp_get_environment_type()`
3. plugin version contains `dev`
4. fallback to `production`

Recommended environment values:

- `production`
- `development`

`debug` remains a separate flag.

Meaning:

- `environment` controls operational mode
- `debug` controls diagnostic verbosity and strictness

Examples:

- `development` disables container compilation
- `production` enables container compilation when possible
- `debug=true` enables stronger diagnostics

Failsafe behavior is allowed for operational recovery only, not for invalid user config.

Allowed failsafe examples:

- compiled container cannot be written, so fall back to live container

Disallowed failsafe examples:

- silently ignoring invalid hook bindings
- silently ignoring invalid enabled module configuration

### Container Compilation

Container compilation should happen on first request in production by default.

`1.0` does not require a compile command.

Reasons:

- WordPress users expect activation-time simplicity
- CI may lack a full WordPress runtime
- service definitions may reference environment-specific classes

A compile command may be added later as optional tooling, but it is not part of the core `1.0` plan.

### WP Desk Paid/Free Separation

Core `wp-init` should remain generic.

It should not publicly encode WP Desk commercial policy in its core API.

WP Desk-specific paid/free concerns should be handled via:

- internal module sets
- meta-packages
- scaffolding
- CI checks in consuming plugins

This keeps the core architecture clean while still allowing WP Desk to enforce packaging discipline in its own ecosystem.

## Target Configuration Shape

### Minimal Config

```php
<?php

return [
    'services' => 'config/services.php',
    'hooks'    => 'config/hooks',
    'modules'  => [],
];
```

### Extended Config

```php
<?php

return [
    'services'    => 'config/services.php',
    'hooks'       => 'config/hooks',
    'modules'     => [
        \WPDesk\Init\Module\RequirementsModule::class => [
            'requirements' => [
                'php' => '7.4',
            ],
        ],
    ],
    'cache_path'  => 'generated',
    'environment' => 'production',
    'debug'       => false,
];
```

### Notes

- rename `hook_resources_path` to `hooks`
- keep `services` as the canonical container definitions key
- keep `cache_path` as an advanced operational setting
- remove root-level WP Desk-specific config keys over time by moving them into module config

## Target Bootstrap Context

Modules and boot internals should receive a small immutable bootstrap context object instead of raw mixed state.

The context should expose:

- plugin file
- plugin path
- parsed plugin header
- slug
- version
- global config
- environment
- debug flag
- module-specific config

The purpose of this object is to:

- avoid fetching bootstrap values back out of the container
- separate bootstrap state from runtime container state
- keep module APIs small and predictable

## Target Abstractions

### Init

Public entrypoint. Responsible for:

- loading configuration
- platform compatibility pre-check
- creating bootstrap context
- creating the internal bootstrap runtime
- booting the plugin

### Module

Explicit opt-in bootstrap behavior.

Responsibilities may include:

- adding service definitions
- adding built-in hook bindings
- providing boot gates
- adapting legacy behavior

Modules are not just implementation detail names for hardcoded built-ins.

### Hookable

Primary unit for attaching WordPress hooks.

A hookable class owns its own hook registration logic.

### Callable Binding

Narrow convenience bridge for one-shot bootstrap actions.

Callable bindings should be validated strictly and documented with narrow rules.

### Boot Gate

Explicit preflight stop mechanism.

Boot order:

1. build context
2. initialize minimal runtime
3. run boot gates
4. if any gate fails, stop boot
5. otherwise continue with normal bindings and modules

### Activation and Deactivation Support

Provide explicit handling for activation and deactivation as special WordPress lifecycle cases.

Do not generalize beyond what WordPress actually needs.

## Simplification Rules

To keep `wp-init` fluid and easy to use, `1.0` should avoid introducing abstractions without clear user value.

Specifically:

- do not introduce a generic registration language
- do not introduce generic grouping abstractions
- do not introduce broad lifecycle frameworks
- do not encode ordinary userland conditional logic into the core model

If a user wants conditional hook registration, that should usually stay inside a `Hookable` class.

## Behavior Rules

### Invalid Configuration

Invalid user configuration should fail loudly and early.

This includes:

- invalid enabled module configuration
- invalid hook binding definitions
- unsupported binding value types
- invalid callable binding signatures

`UnknownDefinition`-style silent no-op behavior must not remain part of the normal runtime path.

### Operational Recovery

Operational failures may recover when recovery is clearly safe.

Examples:

- unable to write compiled container, so use live container
- optional package absent for a module that is not enabled

Operational recovery must not hide broken enabled behavior.

### Callable Validation

Callable bindings remain supported, but must have strict rules and useful error messages.

At minimum, the library should define what callable signatures are allowed and reject unsupported shapes explicitly.

## Documentation Requirements for 1.0

Before `1.0`, docs must be aligned to one coherent mental model.

Required docs:

- canonical README example
- canonical config example
- module registration and module config explanation
- hook binding explanation
- callable binding explanation
- boot gate explanation
- activation/deactivation support explanation
- legacy migration guide
- environment/debug/compilation behavior explanation

Outdated docs that describe old behavior must be removed or rewritten.

## Test Requirements for 1.0

`1.0` requires real coverage of the intended public behavior.

Minimum required test coverage:

- `Init::setup(...)->boot()` happy path
- invalid binding config fails with useful error
- boot gate success path
- boot gate stop path
- hook ordering behavior
- callable binding validation behavior
- production compilation behavior
- fallback when compiled container cannot be written
- module enablement behavior
- legacy module behavior
- activation/deactivation behavior

Tests should validate the supported API, not stale internal shapes.

## Implementation Plan

### Phase 1. Stabilize the contract

- decide and enforce public vs internal API
- remove PHP 8-only syntax from runtime code
- define the target config schema
- define environment and debug behavior

Verify:

- runtime code is PHP `7.4` compatible
- one short document can explain the public API

### Phase 2. Introduce modules cleanly

- replace `Extension` concept with `Module`
- move module config into the `modules` map
- introduce bootstrap context
- remove container-coupled bootstrap APIs where possible

Verify:

- modules are explicit
- module config and enablement live in one place

### Phase 3. Harden boot behavior

- add explicit boot gate abstraction
- replace stoppable binder logic with gates
- tighten binding validation
- keep callable bindings but validate them strictly

Verify:

- invalid config fails early
- stop-boot behavior is understandable and tested

### Phase 4. Handle lifecycle edges

- add explicit activation/deactivation support
- isolate legacy support into a dedicated module
- remove global singleton-like legacy behavior where possible

Verify:

- lifecycle behavior is explicit and testable
- legacy support is isolated

### Phase 5. Tighten operations

- keep first-request production compilation
- preserve safe fallback when compilation cannot be written
- finalize environment/debug behavior

Verify:

- production and development behavior are deterministic
- compilation fallback does not hide invalid config

### Phase 6. Rewrite docs and examples

- update README
- rewrite config docs
- write module docs
- complete legacy migration doc
- update examples and fixtures

Verify:

- docs and tests describe the same system

### Phase 7. Prepare 1.0 release

- remove stale internals from the implied public contract
- close remaining doc gaps
- close remaining test gaps
- prepare migration notes from `0.10.x`

Verify:

- behavior is coherent
- public surface is intentionally small
- no critical ambiguity remains around boot, modules, gates, and compilation

## Explicit Non-Goals for 1.0

- becoming a full application framework
- modeling every WordPress lifecycle event generically
- replacing ordinary plugin code with framework DSL
- exposing WP Desk-specific commercial concepts in the core architecture

## Summary

The `1.0` direction is:

- small public API
- explicit modules
- module-owned config
- hookables as the primary integration unit
- callables as narrow bridges
- explicit boot gates for hard-stop conditions
- explicit activation/deactivation handling
- generic core, WP Desk-specific policy outside core
- PHP `7.4` runtime compatibility

This is the target shape that should guide implementation decisions from this point on.
